# MemoryWeaver 完整建议总结（v0.4 → v0.4.1 → 后续路线）

## 一、项目背景

MemoryWeaver 是一个反馈校准型记忆 Harness，SDK v0.2.0，核心原则是 **LLM proposes, Harness judges**。三层记忆架构（Layer 1 候选 / Layer 2 激活 / Layer 3 Pattern）、四种极性（positive / negative / neutral / ambiguous）、源门控防污染、三级冲突检测（SILENT / WARN / BLOCK）。v0.4 引入了 LLM GraphProposal——让 LLM 为 tag 之间提议候选图连接，以提升检索召回率。

---

## 二、v0.4 实验结果（DeepSeek v4 pro）

### 关键数据

| 指标 | 值 |
|---|---|
| proposals 总数 | 22 |
| gold 匹配数 | 2 |
| wrong link rate | 0.9091 |
| accepted / pending / rejected / quarantined | 7 / 3 / 0 / 12 |
| reject rate | 0.0 |
| 人审需求 | 15 (68%) |

### 检索对比

| Arm | Memory Recall@10 | Candidate Reduction | p95 ms |
|---|---|---|---|
| no_graph | 0.5417 | 0.9125 | 0.0585 |
| manual_graph | 0.8334 | 0.8187 | 0.0993 |
| rule_graph | 0.8334 | 0.8 | 0.1078 |
| deepseek_proposal | 0.9167 | 0.7812 | 0.1385 |

### 暴露的问题

1. **错链率 90.91%** — 22 个 proposal 中 20 个被判定为 wrong，部分原因是 gold edge 使用严格三元组匹配
2. **reject rate = 0** — ReviewPolicy 没有明确拒绝能力
3. **evidence attached ≠ evidence supports** — evidence coverage 1.0 但 wrong link rate 0.9091
4. **DeepSeek 进入在线实时路径** — 延迟从 0.06ms 涨到 0.14ms
5. **candidate-level pollution** — LLM 即使没有权限写 verified memory，也能通过生成大量低质量候选关系增加检索噪声、延迟和审核成本
6. **graph expansion 在小数据上反而更慢** — 全量扫描更便宜

## 三、设计讨论与决策

### 3.1 核心架构变更

LLM 从实时参与者降为离线维护者：

```
离线 batch: new memories → LLM proposal → Harness review → accepted edges → write to graph
在线 query: tag search → rule graph → accepted edges only → rerank → answer
```

### 3.2 新增模块

1. **ProposalBudgetGate** — 限制 LLM proposal 触发条件、数量和成本
2. **EvidenceSupportCheck** — 区分 evidence 是否真正支持关系（supports_exact / partial / does_not_support / contradicts / insufficient_evidence）
3. **GraphExpansionPolicy** — 控制图谱扩展跳数和策略
4. **RiskBasedRelationPolicy** — 高风险 relation 自动拒绝或隔离
5. **OfflineProposalBatchRunner** — 离线批处理调度

### 3.3 BudgetGate 触发条件

```
1. 默认不调用 LLM proposal
2. manual/rule graph recall 不足时才调用
3. query 是新 topic
4. tag expansion 找不到足够 candidate
5. 每个 query 最多 N 个 proposals（动态调整）
6. 高风险 relation 不允许自动 accept
7. wrong link rate 超阈值后自动降级 provider
8. latency 超阈值后禁用 LLM proposal
9. online path 不调用 LLM proposal（硬规则，不可绕过）
10. 加入 cost_ceiling、per_batch_cap、panic_stop 硬性安全天花板
```

### 3.4 Prompt 升级为可证伪关系格式

新增字段：`why_link`、`why_not_link`、`required_evidence`、`relation_strength`、`should_accept`。强制 LLM 输出反证和不确定性，而非努力生成边。

### 3.5 原则升级

```
原来是：LLM proposes, Harness judges.
升级为：LLM proposes only under budget; Harness judges and throttles.
```

## 四、v0.4.1 实现结果

### 4.1 落地内容

- LLM GraphProposal 固定为 offline candidate generator
- online path 禁止调用 LLM proposal（硬规则）
- 新增 ProposalBudgetGate、EvidenceSupportCheck、GraphExpansionPolicy、RiskBasedRelationPolicy、OfflineProposalBatchRunner
- prompt 升级为 v0.4.1（可证伪关系格式）
- review 区分五种 evidence support 状态
- accepted edge 才能作为在线 graph substrate
- 136 个测试通过（v0.4 为 113 个）

### 4.2 关键指标变化

| 指标 | v0.4 | v0.4.1 | 变化 |
|---|---|---|---|
| proposals | 22 | 4 | ↓82% |
| online LLM calls | 多次 | 0 | 完全消除 |
| reject rate | 0.0 | 0.5 | 拒绝能力从无到有 |
| accepted wrong link rate | - | 0.0 | 零错误接受 |
| accepted edges | 7 | 0 | 保守策略生效 |
| evidence support: exact/partial/unsupported | 未区分 | 0.5/0.0/0.5 | 分类机制落地 |

### 4.3 检索对比

| Arm | v0.4 Recall@10 | v0.4.1 Recall@10 |
|---|---|---|
| no_graph | 0.5417 | 0.5417 |
| manual_graph | 0.8334 | 0.8334 |
| rule_graph | 0.8334 | 0.8334 |
| llm_offline_proposal | 0.9167 | 0.5417 |

llm_offline_proposal_graph recall 回落到 no_graph 水平，因为 accepted edges = 0。

### 4.4 正确定性

v0.4.1 是一次成功的 **gate machinery smoke test**，不是一次成功的 LLM graph improvement experiment。

**证明了：**
- online/offline boundary 干净拆分（online_llm_call_count = 0）
- reject 能力存在（reject rate 0.5）
- accepted wrong link rate = 0.0
- ReviewPolicy 不再只 pending，也能 reject
- EvidenceSupportCheck 开始区分 supports_exact 和 does_not_support

**尚未证明：**
- LLM proposal 能为 online retrieval 带来增量 recall
- EvidenceSupportCheck 在真实 LLM 输出下可靠
- LLM proposal 能减少人工建边成本
- graph retrieval 在更大规模下优于 full text scan

## 五、剩余风险

### 5.1 EvidenceSupportCheck 的 supports_exact 可能过松

local provider 生成的 `codex_subscription_failed → codex_cli` proposal，绑定不相关 evidence 却被标记为 `supports_exact` score 0.9。真实 LLM 会利用这种宽松匹配"聪明地乱绑 evidence"。需要人工标注 20-30 对 proposal-evidence 跑分类准确率。

### 5.2 Manual graph 已覆盖 83% recall

40 条记忆 fixture 下，LLM 最多从剩余 17% 里抢。记忆池不够大时，缺失的 recall 可能不是图不够而是记忆本身不足。

### 5.3 Review cost per accepted edge

真实 LLM 场景下，cost = 人审次数 / accepted edges。如果高于手动建边成本，LLM proposal 在经济上不成立。

### 5.4 Pending 队列无限增长

需要 proposal_age、review_deadline、stale_after_batches、auto archive 机制。

## 六、推荐路线

### v0.4.2：真实 LLM provider 压测

- 切换回 DeepSeek（或 Claude/GPT）
- 完整链路：offline batch → LLM proposal → EvidenceSupportCheck → Harness review → accepted edges → online retrieval
- **核心目标**：accepted edge count > 0 且 accepted wrong link rate ≈ 0 且 recall > no_graph 且 online_llm_call_count = 0

### v0.4.2b：EvidenceSupportCheck 校准

- 人工标注 20-30 条 proposal-evidence 对的真实 support 状态
- 计算 EvidenceSupportCheck 分类准确率
- 如果 supports_exact precision < 0.7，先修 checker

### v0.4.3：Pending 生命周期 + mw review

- proposal_age、review_deadline、stale_after_batches
- N 轮无新 evidence → archive
- 新 evidence 支持 → re-review
- 冲突 evidence → reject/quarantine
- `mw review --json` CLI 落地

### v0.5：规模测试 + mw doctor / trace

- 数据规模 100 / 500 / 1000 / 5000 memories
- 对比所有 graph arm 的 latency scaling、candidate reduction、recall retention
- `mw doctor` 和 `mw trace` 落地——把实验验证从手工 JSON 检查变成一条命令

### v0.6：任务级实验

- No memory vs RAG over logs vs MemoryWeaver
- 指标：steps-to-success、repeated error reduction、path reuse rate、tool error rate、memory activation accuracy

### v0.7：Alpha CLI + Bundle

参考 harness-forge 做完整 CLI workflow：

```bash
mw init / doctor / validate / review / trace
mw memory / evidence / pattern / graph
mw export / import (bundle)
mw decision-log
```

### v0.8：离线 Sentinel

```bash
mw sentinel once/run/status
```

离线 watcher：发现新 memory → 检测 stale pending → 生成 offline proposal batch → 不进入在线路径。默认 LLM token budget = 0。

## 七、学术价值

**candidate-level pollution（候选层污染）** — 即使 LLM 没有权限写入 verified memory，仍可通过生成大量低质量候选关系增加检索噪声、延迟和审核成本。这比单纯说"LLM 不能直接写 memory"更深入，是值得记录的论文观察。

## 九、harness-forge 学习建议

harness-forge（[GitHub](https://github.com/ldilov/harness-forge)）是一个面向 AI coding agent 的 runtime harness。**不应该并入 MemoryWeaver，而应该作为工程化 Harness Runtime 参考模板，选择性吸收。**

MemoryWeaver 的核心是"经验记忆与信任边界"；harness-forge 的强项是"工程化 Harness 运行时与操作界面"。

### 最值得吸收的 8 个设计

1. **`.hforge/` 双层可见性结构** — 可见桥接层（AGENTS.md / .agents/）供 agent 发现，隐藏 canonical 层（`.hforge/`）保存权威运行时内容。MemoryWeaver 应引入 `.memoryweaver/`（隐藏）+ `AGENTS.md`（自动生成的可见摘要），防止 agent 自引用循环。

2. **CLI 命令体系** — `mw doctor`（workspace 健康检查）、`mw review --json`（审核状态）、`mw trace`（检索决策路径）、`mw decision-log`（裁决记录）。其中 `mw doctor` 和 `mw review --json` 应提前到 v0.4.3/v0.5，提升实验迭代速度。

3. **Review / Decision Log** — 人类可读的裁决追溯：为什么 accept / reject / pending？哪个 evidence 支持？哪个 policy 版本裁决？影响哪些 memory/pattern？——对应 `mw proposal explain` / `mw edge lineage` / `mw pattern lineage`。这对论文可审计性很重要。

4. **Sentinel / Watcher** — Always-on watcher，默认不花 token、不自动改东西。监控 new memories / evidence / graph drift，触发 offline proposal batch，加入 proposal budget / daily token budget / panic-stop / provider cooldown / pending aging。`signal → approval → action → sandbox` 执行链直接映射到 MemoryWeaver 现有架构。

5. **Approval chain / Side-effect ledger** — SHA-256 chained approval record + 可回滚。对应 GraphEdgeApproval / MemoryPromotionApproval / PatternStabilizationApproval / PolicyChangeApproval。

6. **Export / Import bundle** — `mw export --bundle` / `mw import` 用于项目经验迁移。只导出 stable policy / accepted graph edges / stable patterns with provenance，不导出 raw events / secrets / .env。

7. **cost ceiling + panic stop** — 对 ProposalBudgetGate 的直接补强：每天 LLM proposal token 总量不超过 N，每个 offline batch 最多 1 次 provider 调用，wrong link rate 超过 0.5 立即停用。

8. **Dashboard 数据结构先记录** — events.jsonl / traces.jsonl / decisions.jsonl / metrics.json，先不做 UI 但保留数据结构，以后再做 dashboard。

### 不建议现在吸收的

1. auto-tuning（系统刚暴露 candidate-level pollution，现在自动调参太早）
2. learned patterns 跨项目自动生效（MemoryWeaver 的经验有强 scope，跨项目迁移必须加 scope check + policy version check）
3. 完整 dashboard（先做 CLI 再考虑 UI）
4. GPL 代码本身（harness-forge 是 GPL-3.0，只学设计不复制代码）

### 三者定位

```
MemoryWeaver Core      → 负责 memory / evidence / graph / policy / pattern
MemoryWeaver Runtime   → 负责 CLI / workspace / doctor / trace / review / ledger / sentinel
MemoryWeaver Research  → 负责 benchmark / validation / paper metrics
```
