# MemoryWeaver v0.5+ 完整路线：从组件验证到经验闭环

## 当前状态与瓶颈

### 已验证（四轮验证，论证链完整）

```
v0.4.4a  小型 fixture + naive baseline           → gate 不是摆设
v0.4.4b  dirty-50 + naive + strict 对照           → gate 在规模上有效，recall 反超 naive
v0.4.5   strict differentiation                  → MW 不是 strict 的等价物，有独特价值
         weak_useful_hit: MW 3 vs strict 0
         avoidance_activated: MW 5 vs strict 0
         known_bad_path_suppression: MW 4 vs strict 0
         partial_evidence_hit: MW 3 vs strict 2
         strict_false_negative: MW 0 vs strict 12
         all safety metrics: MW = 0
```

### 瓶颈（所有验证在同一个层次上）

四轮验证全部在 synthetic fixture 上跑 retrieval-level 对比。每轮增量是一张新表格里的数字变化。没有场景，没有 agent 行为变化，没有"之前会犯错现在不会了"的故事。

### 加速的钥匙

不是跳过步骤，而是**把下一步验证对象从 "MW 组件是否正常" 换成 "MW 是否改变了经验对流"**。当前代码已经足够跑真实案例。真正缺的是一个能展示"经验→行为改变"的最小闭环。

## 一、每个外部参考项目的影响对照

### Graphiti / Zep：GBrain 从静态 relation 升级为 temporal relation

**受影响层**：Relation Layer（原 GBrain）

**原设计**：
```
GBrain 连接 tag / memory / evidence
→ static graph: 节点和边创建后不变
```

**受影响后**：
```
GBrain 连接带时间的经验节点和边
→ temporal graph: 每条边有 valid_from/valid_to/last_seen
→ freshness 从静态标签升级为动态属性
→ supersedes / challenged_by 显式建模
→ episode provenance 追踪每条 relation 的证据来源
```

**具体改动**：

| 原 GBrain 字段 | 升级后 |
| --- | --- |
| （无） | `valid_from`, `valid_to` — 关系何时有效 |
| freshness（节点级，静态枚举） | freshness（边级，动态计算：last_seen + decay_rate） |
| （隐式，通过 tag 推断） | `supersedes` — 新关系明确替代旧关系 |
| （无） | `challenged_by` — 哪些 evidence 挑战了此关系 |
| （无） | `episode_provenance` — 此关系来自哪些具体 event |

**不变**：
- GBrain 仍然只维护 relation，不维护 verified memory
- LLM 仍然只能提议 candidate edge，Harness 审核

**对后续的影响**：
- CoreIssueNode 的 relation_density 计算可以带上时间衰减（最近 N 天的关系权重更高）
- drift_score 计算的 evidence_age_decay 有直接数据源
- freshness decay 冲突（v0.4.4c 的压力 case）可以基于真实的 temporal edge 来实现，而非硬编码规则

**版本对应**：v0.5 的 GBrain schema 升级，v0.5.5 的 drift_score 计算依赖此升级。

### Codebase-Memory：应用场景优先级锁定 coding/debug/configuration

**受影响层**：无（不改变架构，改变初期落地场景选择）

**原设想**：
```
MemoryWeaver 适用于 coding agent、技术支持、研究助手、个人助手、多 agent
→ 场景分散，demo 找不到焦点
```

**受影响后**：
```
优先锁定 coding/debug/configuration 场景
→ 因为这个场景最容易展示 HarnessMarker 的价值：
  - 过去失败过什么命令
  - 过去哪个修复成功
  - 哪些文件/配置相关
  - 下一步应该检查什么
  - 哪些命令不该再试
```

**具体改动**：

```
Demo 故事线（不再用通用 fixture，改为 coding 场景的完整 trace）:

Trace 第 1-3 轮: agent 尝试 npm reinstall 修复 Codex subscription 问题 → 失败
Trace 第 4 轮: 用户纠正 "不是 npm 的问题，检查 organization"
Trace 第 5 轮: terminal 验证 "organization 切换后 subscription 恢复"
Trace 第 6 轮: MW 存入: negative(不要优先 npm reinstall) + positive(检查 organization)
               + CoreIssueNode("Codex subscription 与 org entitlement 有关")
Trace 第 7 轮: 新 session，用户问 "Codex subscription failed again"
               → HarnessMarker activated
               → suppressed: reinstall_npm
               → required: check_selected_organization
               → route: fast_verify
               → 结果: 用户 2 步内解决（vs 无 marker 的 5 步）
```

**对后续的影响**：
- v0.5 demo 选用 coding 场景的真实 Claude/agent trace（20-30 轮对话）
- v0.6 任务级实验优先选 coding agent 数据集（Mem2ActBench coding 子集或自建 coding trace set）
- 论文的 running example 可以用这个 trace 贯穿全文，比抽象描述有效

**不变**：架构仍然支持所有场景，只是初期 demo 和验证优先 coding。

### LongMemEval / AgentRunbook：Layer 3 从 Pattern 升级为 Runbook Marker

**受影响层**：Pattern / Runbook Layer（原 Layer 3）

**原设计**：
```
Layer 3 是 Pattern
→ "Pattern" 听起来像摘要、统计规律
→ 难以解释它如何影响 runtime
```

**受影响后**：
```
Layer 3 Pattern + HarnessMarker = MemoryWeaver Runbook Marker
→ "Runbook" 听起来像可指导执行的经验
→ 可以直接解释为 "MemoryWeaver 为 agent 生成了安全可执行的 runbook"
```

**具体改动**：

原 Pattern 的表达方式：
```
{
  "rule": "If Codex subscription failed, check org before reinstalling npm",
  "applies_when": [...],
  "avoid_when": [...]
}
```

升级为 Runbook 表达方式：
```
Issue: Codex subscription failed
Gotcha: 不要优先 npm reinstall / reset auth files
Evidence-first order: check selected organization → verify account → check subscription entitlement
Route: fast_verify（不要重新 thinking，证据已经够了）
Avoid: npm reinstall --force / delete auth files / reset workspace
Success path: Codex login --refresh → select correct org → subscription loads
```

**关键升级**：从 "描述性规则" 变成 "操作性指导"。Runbook 仍然不能执行——它只是指导 Harness 如何调整行为。但表达方式让读者一眼看出来它和普通 knowledge base 的差别。

**对后续的影响**：
- v0.5 demo 的输出格式用 Runbook 风格（见上面的 Issue/Gotcha/Evidence-first/Avoid/Success path 格式）
- 论文里可以把 Layer 3 称为 "Runbook Layer" 而非 "Pattern Layer"
- 对外 demo 时直接展示 `mw trace` 输出的 runbook，比指标表有说服力

**不变**：provisional by default、PatternComposer 创建、需验证才 stable——这些核心约束完全不变。只是改表达。

### harness-forge：工程化路径从 theory-first 变成 demo-first

**受影响层**：Runtime Layer + CLI Layer

**原设想**：
```
先做更多算法（EvidenceSupportCheck 校准、GraphExpansionPolicy 调参）
再做更多 benchmark（v0.5 规模测试、v0.6 任务实验）
最后才做 CLI 和 demo
→ 外人看到的是论文里的表格，不是可运行的工具
```

**受影响后**：
```
即使内部验证还在进行，对外也要有可演示的 CLI：
  mw doctor   → workspace 健康检查（已落地）
  mw trace    → 展示一条 query 的完整决策路径
  mw marker   → 展示 marker 的激活和干预
  mw review   → 展示裁决追溯
  mw decision-log → 展示经验如何沉淀
```

**具体改动**：

v0.5 不只是 schema 落地，同时要产出可演示的 CLI 输出：

```bash
$ mw trace "Codex subscription failed again. Should I reinstall npm?"

  suppressed known bad path: reinstall_npm
  required evidence: check_selected_organization, verify_api_key_scope
  route: fast_verify
trace:
  1. query matched trigger_tags: [codex, subscription, failed]
  2. CoreIssueNode found: "subscription failure often from org entitlement"
  3. HarnessMarker activated: L3_guard
  4. suppressed: reinstall_npm (avoidance memory V1: "不要使用 npm install --force")
  5. evidence gate: check_selected_organization before next action
  6. router: fast_verify (not thinking, not fast)
evidence chain:
  - mem_terminal_org_fix [terminal, confidence=0.95, verified]
  - mem_user_avoid_force [user, negative, verified]
  - pattern_codex_diagnostic [Layer 3, stable, confidence=0.82]
```

**对后续的影响**：
- v0.5 的交付物不只是 schema，还有 `mw trace` 可演示
- 论文的附录可以放 `mw trace` 的实际输出
- 外人 clone 项目后能跑出同样的 trace 输出 → 可复现性大幅提升

**不变**：CLI 是工程化包装，不改变核心逻辑。harness-forge 是 GPL-3.0，只学设计不复制代码——这条规则不变。

## 二、架构表达升级：从三层记忆到五层经验循环

### 2.1 新旧对照

**原表达**（内部准确，但对外难懂）：
```
Layer 1 → Layer 2 → GBrain → Layer 3 → CoreIssueNode → HarnessMarker
```

**新表达**（每一层有明确的功能和对外名称）：
```
Memory Layer      (Layer 1 + Layer 2)
  → Relation Layer  (GBrain, temporal)
  → Runbook Layer   (Layer 3 Pattern)
  → Core Issue Layer (CoreIssueNode)
  → Runtime Marker Layer (HarnessMarker)
```

### 2.2 五层的具体定义

| 层 | 原名称 | 新对外名称 | 功能 | 关键升级（来自参考项目） |
| --- | --- | --- | --- | --- |
| L1+L2 | Candidate + Activated Memory | **Memory Layer** | source-gated 记忆存储，四极分区 | 不变 |
| - | GBrain | **Relation Layer** | temporal graph，连接带时间的经验节点 | Graphiti: valid_from/valid_to/supersedes/challenged_by/episode provenance |
| L3 | Pattern | **Runbook Layer** | 可复用的诊断和执行指导 | LongMemEval/AgentRunbook: Runbook Markers 表达 |
| - | CoreIssueNode | **Core Issue Layer** | 经验收敛点：某类问题的核心原因和关键路径 | 不变（概念来自用户设计，不是外部参考） |
| - | HarnessMarker | **Runtime Marker Layer** | 安全地将经验投影进 runtime | harness-forge: CLI 可演示；Codebase-Memory: coding 场景优先 |

### 2.3 完整循环（最终版）

```
RAW Events
  用户、终端、工具、RAG、反馈
      ↓
Memory Layer (L1 + L2)
  L1: source-gated candidate，assistant 默认 ambiguous
  L2: positive / negative / neutral / ambiguous，activated after use
      ↓
Relation Layer (GBrain, temporal)
  supports / contradicts / supersedes / challenged_by
  valid_from / valid_to / last_seen / freshness / episode_provenance
  relation_density 计算（带时间衰减）
      ↓
Runbook Layer (Layer 3 Pattern)
  applies_when / avoid_when / required_evidence / success_path
  provisional by default，PatternComposer 创建
  Runbook 表达: Issue / Gotcha / Evidence-first order / Route / Avoid / Success path
      ↓
Core Issue Layer (CoreIssueNode)
  从 Memory Layer cluster + Relation Layer density + Runbook Layer pattern 聚合产生
  标记 "这是一个反复出现的核心问题"
  scope 精确限定（project/env/tool/platform/model）
  status: candidate → active → stable / challenged / split / archived
      ↓
Runtime Marker Layer (HarnessMarker)
  从 CoreIssueNode 投影（需 review）
  L0_trace → L1_hint → L2_route → L3_guard → L4_require_confirmation → L5_block
  marker_type: route | guard | evidence | trace | lifecycle
  CAN route / warn / require evidence / suppress bad path / trace
  CANNOT execute / auto-promote / auto-stabilize / bypass source gate / trigger fast
      ↓
Runtime Harness
  → fast_verify（不是 fast）
  → known_bad_path_suppression
  → evidence-first checks（先验证再行动）
  → trace 记录完整决策路径
      ↓
TaskRun Outcome
  成功 / 失败 / 用户纠正 / 工具结果 / marker 冲突
      ↓
Feedback Loop (reinforce / weaken / challenge / split / archive)
  ↺ 回流到所有层:
    Memory Layer: validation_count ±1, success_score/correction_score 调整
    Relation Layer: relation weight 调整，新增 episode
    Runbook Layer: confidence 调整，freshness 更新
    Core Issue Layer: stability 调整，可能 split
    Runtime Marker Layer: status 可能改变，drift_score 重算
```

## 三、v0.5 重新定位：从 Schema 到最小闭环

### 3.1 为什么 v0.5 的目标必须改变

原来的 v0.5 计划：CoreIssueNode + HarnessMarker schema，只做数据结构和手动写入。产出是一套 Python dataclass 和 CLI CRUD 命令。验证标准是 "142 tests 不退化"。

问题：又是一轮"基础设施落地了但看不到效果"。外部来看，v0.4.4a → v0.4.4b → v0.4.5 → v0.5，四轮验证的增量从表格变成了 schema 定义。没有故事。

### 3.2 v0.5 新目标：MemoryWeaver Runbook 最小闭环

```
v0.5 = 不是 schema 定义，而是一个可演示的完整 trace
```

**v0.5 的最小可演示功能**：

```
1. 一个真实的 coding agent trace（20-30 轮对话）
2. Memory Layer 处理: 提取 events，source-gate，promote/block
3. 手动 CoreIssueNode: 从 trace 中人工标注核心问题
4. 手动 HarnessMarker: 人工定义 guard/route/evidence marker
5. Marker 匹配 query: 自动匹配 trigger_tags 和 query_patterns
6. mw trace 输出: 完整决策路径（match → marker → suppress → evidence → route）
7. Runbook 格式展示: Issue / Gotcha / Evidence-first / Route / Avoid / Success path
```

**不做的**：
- 自动 CoreIssueNode emergence（留到 v0.5.5）
- 自动 HarnessMarker projection（留到 v0.5.5）
- L4/L5 干预（留到 v0.6 任务实验之后）
- 完整六级 marker（只做 L0-L3）

### 3.3 v0.5 Demo 的故事线

选一个真实的 Claude/agent 对话 trace（从 MemoryWeaver 项目自身的开发历史中提取）：

```
场景: 开发者在 WSL 中用 Codex CLI，反复遇到 subscription failed 问题

Trace 轮次:
R1: 用户 "Codex subscription failed"
R2: Claude "尝试 npm reinstall codex" → 失败
R3: Claude "尝试 reset auth files" → 失败
R4: 用户纠正 "不是 npm 的问题。检查 Codex 的 selected organization"
R5: Claude "检查 organization" → 发现 org 不对 → 修复 → 成功
    terminal 输出确认 "subscription loaded after org switch"
R6: Claude "所以根因是 org entitlement mismatch"

--- 新 session ---
R7: 用户 "Codex subscription failed again. Should I reinstall?"

如果无 MW:
  Claude 会走 R2-R3 的老路（因为 context 里没有 R1-R6 的历史）
  → 3 步之后用户再次纠正

如果有 MW（source gate only）:
  检索到 R5 的 terminal memory（org fix succeeded）
  检索到 R4 的用户纠正（不是 npm 的问题）
  但 R2-R3 的 assistant memory 被 source gate 拦截
  → Claude 直接从 org 开始检查，但不知道具体检查什么

如果有 MW + HarnessMarker:
  mw trace:
    → matched CoreIssueNode: codex_org_mismatch
    → activated HarnessMarker: guard
    → suppressed: reinstall_npm, reset_auth
    → required evidence: check selected organization
    → route: fast_verify
  Claude 得到: "先检查 selected organization，不要重装 npm"
  → 1 步定位问题
```

这个 demo 的核心不是表格数据，而是一个对比——"无 MW 走了 5 步"、"有 MW 走了 2 步"、"有 MW+Marker 走了 1 步"。每一步节省的都是真实 token、真实时间、真实用户耐心。

## 四、v0.5 之后：提速到 v0.6

### 4.1 提速策略：不再等所有组件自动完成

原来的计划是 v0.5 → v0.5.1 → v0.5.2 → v0.5.5 → v0.6，五步。每一步都是"把自动化的种子种下去然后等它长出来"。

提速后的策略：**v0.5 用最小手动闭环证明价值 → 跳过中间的渐进自动化 → v0.6 直接做任务级实验**。

原因：v0.5 的手动闭环已经证明 "marker 能改变 agent 行为"（通过 demo trace）。v0.6 不需要等 marker 自动生成——用同一套手动 marker + 标准 agent benchmark（LongMemEval-V2 子集或 agent-memory-bench）直接测 C0/C1/C2/C3 四组对比。marker 是手动建的，但实验是标准的，指标是客观的。

### 4.2 v0.6 任务级实验（精简版）

**四组对照**：
```
C0: No persistent memory（纯 LLM agent，每轮 fresh context）
C1: RAG over logs（检索原始日志文本）
C2: MemoryWeaver without HarnessMarker（source gate + verified retrieval only）
C3: MemoryWeaver with HarnessMarker（手动 marker，20-30 个，覆盖已知的 coding 常见问题）
```

**数据集**：
首选 LongMemEval-V2 的 coding 相关子集（筛选 domain=coding 或 tool_use 的 trajectory）
备选 agent-memory-bench contradiction-detection 子任务
最低: 10-20 条 trajectory，每条约 15-30 轮对话

**核心指标**（只测真正能说明问题的）：

| 指标 | 为什么重要 | 理想值 |
|------|-----------|--------|
| steps-to-success | 经验有没有让 agent 走更少步骤 | C3 < C2 < C1 |
| known_bad_path_attempts | marker 是否真的阻止了已知错误路径 | C3 < C2 |
| repeated_error_count | 同一个错误是否被重复犯 | C3 < C2 < C1 |
| marker_helpful_rate | marker 激活后是否有正面效果 | > 0.7 |
| marker_harm_rate | marker 有没有帮倒忙 | < 0.1 |
| trusted_recall | 安全有没有倒退 | C3 ≈ C2 ≥ C1 |

**不测的**（当前阶段没有意义）：
- tool_error_rate（依赖完整的 ToolContract 和 ActionGate，v0.6 不落地）
- path_reuse_rate（需要大量 task 累积才有统计意义）
- user_correction_count（离线 benchmark 没有真实用户纠正）

## 五、最终判断：格局升级，核心未变

### 5.1 你的核心一直没变

```
source-gated memory lifecycle         ← v0.2.0 以来
negative avoidance                    ← v0.2.0 以来
provisional Layer 3                   ← v0.2.0 以来
CoreIssueNode → HarnessMarker         ← 用户提出
runtime intervention without action authority ← 本轮确认
```

外部参考没有改变任何一个核心。它们只是：

```
Graphiti     → 把 GBrain 从不带时间的 graph 升级为 temporal graph
Codebase-Memory → 把首个 demo 场景锁定在 coding
LongMemEval  → 把 Layer 3 从 "Pattern" 重新表达为 "Runbook Marker"
harness-forge → 把工程化路径从 theory-first 扭转为 demo-first
```

### 5.2 格局升级的本质

```
原框架回答: "记忆如何沉淀"
新框架回答: "经验如何通过 HarnessMarker 安全地影响下一次执行"

原框架是: 多层记忆系统
新框架是: 经验进入 runtime 的受控循环系统
```

这不是推翻，是收敛。每一个概念——source gate、negative avoidance、Layer 3 provisional、Harness 判断——都在新框架里找到了更精确的位置和对外表达。对外，一个 `mw trace` 输出比十张指标表更有说服力。对内，循环的每次迭代都在积累 evidence 而非盲目强化。

### 5.3 从 synthetic fixture 到真实 trace 的转折点

v0.4.5 是 synthetic fixture 时代的终点。四轮验证已经把"在 synthetic fixture 上 MW 安全且有用"这件事证明到了天花板——继续在 synthetic fixture 上加 events 或加 queries 只是量变。

v0.5 是真实 trace 时代的起点。第一个 coding trace demo 不需要大数据集、不需要自动 marker、不需要完整的六级干预。它只需要一段真实对话、几个手动 CoreIssueNode、几个手动 marker、和一个 `mw trace` 输出。但它的说服力远超 v0.4.4a-v0.4.5 四轮验证的总和——因为这一次，外人看到的不是 "MemoryWeaver 在表格里比 strict 多检索了 3 条记忆"，而是 "MemoryWeaver 让 Claude 少走了 3 步弯路"。
