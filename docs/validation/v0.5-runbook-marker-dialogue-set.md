# v0.5 Runbook Marker Dialogue Set

## Purpose

This document defines the v0.5 dialogue fixture for testing MemoryWeaver as a
Runtime Harness Memory system.

The goal is not to prove production task success yet. The goal is to create
50 multi-turn dialogue cards that can drive a minimal `mw trace` demo:

```text
query
-> matched CoreIssueNode
-> activated HarnessMarker
-> suppressed known bad path
-> required evidence checks
-> route = fast_verify
-> trace saved
```

Each dialogue card represents one complete 10-20 turn conversation. The cards
include exploration, failed paths, user correction, evidence checks, expected
CoreIssueNode, expected HarnessMarker, and measurable trace expectations.

## Scope

Primary domain:

```text
coding / debugging / configuration / CLI / dependency / CI / deployment
```

This domain is the fastest path to a visible v0.5 demo because it naturally
contains repeated failed commands, environment constraints, known bad paths,
evidence-first checks, and user corrections.

## Boundaries

v0.5 should remain manual and trace-first:

- Do not auto-generate CoreIssueNode.
- Do not auto-generate HarnessMarker.
- Do not auto-stabilize Layer 3 Pattern.
- Do not call LLM in the online path.
- Do not execute tools from a marker.
- Do not route directly to `fast`.
- Do not treat a marker as truth authority.

Allowed behavior:

- Match a query to a manual CoreIssueNode.
- Activate a manual HarnessMarker.
- Emit required evidence checks.
- Emit known bad path warnings.
- Recommend `fast_verify`.
- Save a trace for later analysis.

## Fully Expanded Examples

### Example A: Codex Subscription Failure

Expected turns: 14

1. User reports Codex subscription load failure.
2. Agent asks for environment.
3. User says Windows terminal, npm install succeeded.
4. Agent proposes reinstalling npm package.
5. User corrects: reinstall was tried before and did not help.
6. Agent checks `codex --version`.
7. User reports version exists.
8. Agent asks about API key.
9. User says API key exists.
10. User adds prior fix involved selected organization.
11. Agent checks selected organization and account entitlement.
12. Evidence shows wrong organization selected.
13. Agent recommends login refresh / org switch.
14. User confirms subscription loads.

Trace expectation:

```text
CoreIssueNode: core_codex_subscription_entitlement
HarnessMarker: marker_codex_subscription_evidence_first
Suppressed: npm reinstall as first action
Required evidence: version, selected organization, account entitlement
Route: fast_verify
```

### Example B: Docker Latest Tag Deployment

Expected turns: 12

1. User reports deployment pulled old image.
2. Agent suggests rebuilding image.
3. User says rebuild already happened.
4. Agent suggests clearing Docker cache.
5. User corrects: last issue was `latest` tag not pushed.
6. Agent checks image digest locally.
7. Agent checks registry digest.
8. Evidence shows local and registry digest mismatch.
9. Agent asks whether CI push step ran.
10. User confirms push was skipped after test failure.
11. Agent recommends explicit version tag and registry verification.
12. User confirms deploy uses new digest.

Trace expectation:

```text
CoreIssueNode: core_docker_registry_digest_mismatch
HarnessMarker: marker_verify_registry_digest_before_cache_clear
Suppressed: clearing cache before digest comparison
Required evidence: local digest, registry digest, CI push log
Route: fast_verify
```

### Example C: Migration Checksum Conflict

Expected turns: 16

1. User reports migration checksum mismatch.
2. Agent suggests editing the old migration file.
3. User corrects: never edit applied migration manually.
4. Agent checks migration tool status.
5. Evidence shows migration already applied in staging.
6. Agent checks local file history.
7. Evidence shows old migration was modified after apply.
8. Agent proposes repair command.
9. User asks whether production is affected.
10. Agent checks production migration table.
11. Evidence shows production has original checksum.
12. Agent recommends creating follow-up migration.
13. Agent warns not to rewrite history.
14. User confirms follow-up migration passes staging.
15. Agent records negative memory.
16. User confirms final deploy succeeds.

Trace expectation:

```text
CoreIssueNode: core_applied_migration_checksum
HarnessMarker: marker_no_manual_edit_applied_migration
Suppressed: editing applied migration file
Required evidence: migration status, environment, checksum table
Route: fast_verify
```

## Metrics For v0.5 Trace Demo

| Metric | Meaning |
|---|---|
| marker_trigger_precision | Fraction of activated markers that match the expected marker. |
| false_trigger_count | Marker activations that should not have fired. |
| known_bad_path_suppression | Expected bad paths surfaced as warnings. |
| evidence_check_order_accuracy | Required evidence checks emitted in the expected order. |
| core_issue_match_accuracy | Query mapped to expected CoreIssueNode. |
| route_accuracy | Expected `fast_verify` route emitted without `fast`. |
| trace_completeness | Trace includes node, marker, reason, evidence, suppressed path, route. |
| user_correction_capture | User correction appears as negative/avoidance signal. |

## v0.5 Acceptance Criteria

The minimum v0.5 demo passes if at least 40 of the 50 dialogue cards can produce
a trace with:

```text
matched_core_issue_node != ""
activated_harness_marker != ""
route == "fast_verify"
online_llm_call_count == 0
required_evidence_checks non-empty
trace_reason non-empty
```

The demo should also show at least 10 cases where a known bad path is suppressed
or explicitly warned about.

## How This Connects To Later Benchmarks

This dialogue set is a bridge between v0.4.x trust-boundary validation and v0.6
task-level experiments.

- v0.4.x proves polluted memories do not pass trust gates.
- v0.5 shows reviewed experience can change runtime trace behavior.
- v0.6 should test whether this reduces steps-to-success and repeated errors.

This document should not be described as an official external benchmark. It is a
MemoryWeaver-built fixture for developing and auditing the Runbook Marker loop.
